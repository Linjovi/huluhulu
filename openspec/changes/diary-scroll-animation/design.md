## Context

当前日记系统使用 Cloudflare D1 数据库存储日记数据，前端通过 `/api/blog/get` 接口获取富文本 HTML 内容并直接渲染。这种方案虽然实现简单，但存在以下问题：

1. **内容展示单调**: 所有内容一次性呈现，缺乏叙事节奏
2. **图文混排受限**: 富文本难以实现复杂的图文布局
3. **交互体验平淡**: 没有利用滚动等用户行为增强沉浸感

项目使用 React + TypeScript 前端，Cloudflare Workers + D1 后端，Tailwind CSS 进行样式处理。

## Goals / Non-Goals

**Goals:**

- 设计支持文本块和图片块的日记内容数据结构
- 实现滚动触发的分段动画显示效果
- 保持现有日记元数据（心情、天气、日期等）不变
- 确保动画性能流畅，支持移动端

**Non-Goals:**

- 不支持视频、音频等其他媒体类型（未来可扩展）
- 不实现复杂的动画序列（如链式动画、时间轴控制）
- 不修改日记的创建/编辑功能（仅改造展示）

## Decisions

### 1. 段落块数据结构设计

**决策**: 数据库保持 `content` 富文本字段不变，前端解析富文本生成段落块

**前端解析策略**:
```typescript
// 富文本 HTML 解析为段落块
interface DiaryBlock {
  id: string;           // 块唯一标识（基于索引生成）
  type: 'text' | 'image';
  content: string;      // 文本内容或图片 URL
  caption?: string;     // 图片说明（仅图片块）
}

// 解析规则：
// - <p>, <div> 等文本标签 → text 块
// - <img> 标签 → image 块（src 作为 content，alt 作为 caption）
// - 其他标签按内容类型判断
```

**理由**:
- 保留现有数据库结构，无需数据迁移
- 富文本编辑体验成熟，内容创作更便捷
- 解析逻辑封装在前端，灵活可控
- 渐进式增强，不影响现有功能

**替代方案考虑**:
- 数据库存储 JSON 块：需要修改数据结构，增加复杂度
- Markdown 格式：需要切换编辑器，改动较大

### 2. 动画触发机制

**决策**: 使用 Intersection Observer API 检测元素进入视口

**理由**:
- 原生 API，无需额外依赖
- 性能优于 scroll 事件监听
- 支持配置 root margin，提前触发动画

**动画参数**:
- 触发阈值: 0.2 (元素 20% 可见时触发)
- 动画时长: 600ms
- 缓动函数: cubic-bezier(0.4, 0, 0.2, 1)
- 初始状态: opacity 0, translateY 30px
- 结束状态: opacity 1, translateY 0

### 3. 数据解析策略

**决策**: 前端实时解析富文本 HTML，无需数据库变更

**解析流程**:
1. API 返回富文本 `content` 字段
2. 前端使用 DOMParser 解析 HTML
3. 遍历子节点生成 `DiaryBlock` 数组
4. 每个块分配唯一 ID（基于节点索引）

**理由**:
- 零数据库变更，降低风险
- 解析开销小，现代浏览器性能充足
- 可随时调整解析规则，无需修改数据

### 4. 图片块处理

**决策**: 图片 URL 存储在块内容中，前端使用标准 img 标签加载

**理由**:
- 项目已使用 Cloudflare Images 或外部图床
- 无需修改现有图片存储方案
- 支持懒加载优化性能

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| 动画在低端设备卡顿 | 使用 CSS transform 和 opacity 实现硬件加速；添加 `prefers-reduced-motion` 媒体查询支持 |
| 大量图片块导致加载缓慢 | 图片使用懒加载；添加骨架屏占位符 |
| HTML 解析性能问题 | 日记内容通常较短，解析开销可忽略；使用高效的 DOMParser API |
| 富文本格式不兼容 | 支持常见标签（p, div, img, h1-h6, br, span 等）；未知标签降级为文本处理 |

## Migration Plan

### 阶段 1: 前端解析器开发
1. 实现 `parseHtmlToBlocks` 工具函数
2. 支持常见富文本标签解析（p, div, img, h1-h6 等）
3. 添加单元测试验证解析逻辑

### 阶段 2: 组件开发
1. 创建 `TextBlock` 和 `ImageBlock` 组件
2. 实现 `DiaryContent` 容器组件
3. 集成滚动动画系统

### 阶段 3: 页面重构
1. 重写 `blog.tsx`，使用新的内容渲染方式
2. 移除 `dangerouslySetInnerHTML`
3. 保留旧版渲染作为 fallback（配置开关）

### 阶段 4: 验证与发布
1. 测试各种富文本内容的解析效果
2. 验证动画性能
3. 灰度发布，观察用户反馈

### 回滚策略
- 前端：通过配置开关可快速切换回旧版渲染

## Open Questions

2. 文本块是否需要支持 Markdown 格式？（建议：基础格式如加粗、斜体）
3. 动画是否需要在用户设置中可关闭？（建议：是，尊重用户偏好）
